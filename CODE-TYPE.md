# 代码规范

## 1. 适用范围

- 本规范适用于本仓库全部源码与配套文档。
- 本项目是协议/算法实现项目，规范重点是模块边界、协议口径、类型说明、统计定义和注释质量。

## 2. 总体原则

- 以 `RFC 8878` 和当前源码行为为准，不以历史文档描述为准。
- 优先做小改动、精确改动，不为了“看起来统一”而引入额外抽象。
- 没有真实 `base/tool` 目录时，不要虚构公共层。
- 同一概念在多个模块重复出现时，文档和代码都要带模块语义区分。

## 3. 模块边界

- `api` 负责对外暴露高层能力，不承载过多底层协议细节。
- `core` 负责基础类型、错误、常量、bitstream 和窗口基础模型。
- `encoder` 负责压缩与写帧，不越权承担外部资源管理。
- `decoder` 负责解压、帧头解析、块解析、滑窗管理和分析器。
- `entropy` 负责 FSE、Huffman、序列表和相关公式。
- `dictionary` 负责字典构建、格式转换、收益计算。
- `test`、`examples`、`cmd`、`benchmark` 不要反向主导生产模块设计。

## 4. 命名规范

- 模块别名统一沿用仓库现状，如 `@zstd_core`、`@zstd_decoder`、`@zstd_encoder`。
- 类型名使用大驼峰，如 `FrameHeader`、`CompressionConfig`。
- 函数名使用下划线风格，如 `create_config_with_level`、`decompress_with_limit`。
- 名称必须表达真实语义，不得把启发式结果命名成“精确值”。
- 如果一个值表示 `original_size / compressed_size`，名称才允许使用 `compression_ratio`。
- 如果一个值表示 `compressed_size / original_size`，应改名为 `size_ratio`、`compressed_ratio` 或文档中显式说明。

## 5. 类型与结构体规范

- 新增结构体前，优先复用现有类型，避免同义重复。
- 同名类型如果必须存在于不同模块，必须在文档中标记“所属模块”和“用途差异”。
- 类型字段应尽量表达单位、方向和来源。
- 字节长度统一使用 `Int`，文档写明单位为“字节”。
- 百分比字段统一使用 `Double`，文档写明单位为 `%`。
- 比率字段统一使用 `Double`，文档写明展示后缀为 `x`。
- 启发式字段必须在文档中标注“启发式 / 估算 / 非严格统计模型”。

## 6. 常量与阈值规范

- 协议阈值必须尽量集中定义，并在注释中标明来源或含义。
- 出现“魔法数”时，应尽量提取为常量，或补充解释注释。
- 涉及协议边界值时，要在文档中写明：
  - 取值范围
  - 单位
  - 超限后的行为
- 当前关键阈值包括但不限于：
  - `131072`：最大块大小，单位字节
  - `134217728`：全局输出限制，单位字节
  - `32768`：字典 ID 非保留最小起点

## 7. 错误处理规范

- 生产代码优先返回结构化错误或带语义的 `Result`。
- 错误消息要能回答三件事：哪里错、在做什么、为什么错。
- 如果已有 `ErrorContext` / `ContextualError` 可复用，优先复用。
- `timestamp` 这类历史命名如果实际不是时间，文档必须注明真实语义，避免误导。
- 对协议保留位、越界大小、截断数据等错误，要优先给出明确报错，而不是沉默兜底。

## 8. 注释规范

- 注释统一使用中文。
- 复杂方法必须有方法级文档注释，至少说明：
  - 作用
  - 输入
  - 输出
  - 关键步骤
  - 边界行为
- 复杂代码块前应补充步骤注释，解释为什么这样做。
- 不写纯废话注释，例如“给变量赋值”。
- 协议字段解析、位运算、表驱动逻辑、窗口复制逻辑必须有足够注释。

## 9. 协议解析与写入规范

- 帧头、块头、字典格式等协议解析必须按 RFC 和当前实现顺序书写。
- 位运算逻辑要明确 bit 位来源和字段含义。
- 小端读写函数要复用现有 helper，不要重复发明。
- `RLE`、`Raw`、`Compressed` 三种块在大小口径上要分开说明。
- 如果实现与标准实现近似但不完全相同，例如简化校验和，必须显式标注“近似实现”。

## 10. 统计口径规范

- 所有统计结果对象必须记录：
  - 字段含义
  - 单位
  - 来源
  - 公式
  - 精度或取整方式
- 当前统一口径：
  - `compression_ratio = original_size / compressed_size`
  - `saved_bytes = compressed_size_without_dict - compressed_size_with_dict`
  - `benefit_percentage = saved_bytes / compressed_size_without_dict * 100`
- 估算值和真实值必须区分命名和文档。
- 浮点数展示时，若做格式化，需注明保留位数和取整策略。

## 11. 测试规范

- 新增协议逻辑、边界判断、公式映射时，应优先补对应测试。
- 测试名称要体现被验证的协议行为或缺陷编号。
- 对历史 bug 回归测试，建议保留问题背景说明。
- 金标样本、错误样本、文本样本应按用途分类存放在 `src/test-data/`。

## 12. 文档同步规范

- 新增或调整公开类型时，更新对应模块 `RESOURCE.md`。
- 新增或调整统一规范时，更新 `CODE-TYPE.md`。
- 新增或调整模块索引时，更新根 `AGENTS.md`。
- 如果代码行为与历史文档冲突，应先修正文档，再视情况修正代码或补测试。

## 13. 推荐开发检查

```powershell
moon run src/cmd
rg "^pub (struct|enum|type|fn)" src
rg "TODO|FIXME|HACK" src
```

## 14. 文档组织要求

- `AGENTS.md` 只保留轻量索引、术语和快捷命令。
- `RESOURCE.md` 存放模块级对象、字段、公式、统计口径和资源说明。
- 不把所有细节堆进根文档，避免 AI 上下文过大。
